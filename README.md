# Робот: управление с ПК (жесты + клавиатура + Wi‑Fi)

Программа на **Python** захватывает изображение с веб-камеры, распознаёт руки через **MediaPipe Hand Landmarker** и отправляет команды на машинку с **ESP8266** по **HTTP** (Wi‑Fi). Дополнительно доступно управление **клавиатурой** (WASD и стрелки).

## Что нужно для работы

| Компонент | Описание |
|-----------|----------|
| **ПК** | Windows (как у тебя), Linux или macOS; установленный **Python 3.10+** (рекомендуется 3.10–3.12 для совместимости с MediaPipe). |
| **Веб-камера** | Подключена к ПК; в коде по умолчанию устройство `0` (`cv2.VideoCapture(0)`). |
| **ESP8266** | С прошивкой из `firmware/esp32_car_drive_server/`, драйвер моторов **L298N** по пинам из `.ino`. |
| **Сеть** | Ноутбук в той же Wi‑Fi сети, что и ESP, **или** ноутбук подключён к точке доступа ESP8266. |

## Установка на компьютер

### 1. Клонирование или копия папки

Скопируй проект в удобное место (например `Desktop\robot`).

### 2. Виртуальное окружение (рекомендуется)

**PowerShell (Windows):**

```powershell
cd C:\Users\ASUS\Desktop\robot
python -m venv venv
.\venv\Scripts\Activate.ps1
pip install --upgrade pip
pip install -r requirements.txt
```

Если политика запрещает скрипты:

```powershell
Set-ExecutionPolicy -Scope CurrentUser RemoteSigned
```

**Без venv** можно просто:

```bash
pip install -r requirements.txt
```

### 3. Зависимости из `requirements.txt`

- `opencv-python` — камера и окно с картинкой
- `mediapipe` — распознавание рук (Tasks API)
- `pyserial` — опционально, если включишь управление по COM-порту

## Модель распознавания рук

Файл **`models/hand_landmarker.task`** должен лежать по пути:

`robot/models/hand_landmarker.task`

Если файла нет, скачай официальную модель **Hand Landmarker** (формат `.task`) с репозитория ассетов MediaPipe, например:

- [Google Storage — hand_landmarker.task](https://storage.googleapis.com/mediapipe-models/hand_landmarker/hand_landmarker/float16/latest/hand_landmarker.task)

Создай папку `models`, положи туда файл и убедись, что имя в коде совпадает: в `main.py` указано `MODEL_PATH = ... / "hand_landmarker.task"`.

## Прошивка ESP8266

1. Установи **Arduino IDE** и поддержку плат **ESP8266** (менеджер плат).
2. Установи библиотеки: **ESP8266WiFi**, **ESP8266WebServer** (обычно входят в пакет ESP8266).
3. Открой `firmware/esp32_car_drive_server/esp32_car_drive_server.ino`.
4. Выбери свою плату и COM-порт, загрузи скетч.

После старта в **Serial Monitor** (скорость **115200**) будет строка вида:

`AP IP address: 192.168.4.1`

Имя Wi‑Fi точки задаётся в коде: `const char* ssid = "Make DIY";` — при желании поменяй.

## Сеть и адрес в `main.py`

Программа шлёт запросы на адрес из переменной:

```python
CAR_WIFI_BASE_URL = "http://192.168.4.1"   # пример для режима AP у ESP8266
```

- Если ноутбук **подключён к точке ESP** — чаще всего IP **`192.168.4.1`**.
- Если ESP и ноутбук в **домашней сети** (ESP в режиме STA или другой схеме) — укажи **реальный IP** ESP в локальной сети (как в твоём `192.168.0.15`, если так настроено).

Без совпадения IP в `CAR_WIFI_BASE_URL` с реальным адресом ESP команды **не дойдут** (в HUD будет `net err`, `timeout` и т.п.).

## Полная таблица команд

Сначала программа вычисляет **логический режим** (`STOP`, `FWD`, `FWD_L`, `FWD_R`, `REV`), затем переводит его в **одну букву** для прошивки (`State`).

| Режим в Python | Параметр HTTP `State` | Действие на машинке (прошивка) |
|----------------|------------------------|--------------------------------|
| `STOP` | `S` | Стоп, отключение ШИМ на ENA/ENB |
| `FWD` | `F` | Оба мотора вперёд |
| `FWD_L` | `G` | Вперёд с поворотом влево (разная скорость сторон) |
| `FWD_R` | `I` | Вперёд с поворотом вправо |
| `REV` | `B` | Оба мотора назад |

**Дополнительные буквы в прошивке** (Python их сейчас не шлёт, но ESP понимает):

| `State` | Функция в `.ino` |
|---------|------------------|
| `L` | Поворот на месте влево |
| `R` | Поворот на месте вправо |
| `J` | Назад + вправо |
| `H` | Назад + влево |
| `0`–`9` | Ступени скорости ШИМ (400…1023) |

Пример запроса в браузере или `curl` (подставь свой IP):

```text
http://192.168.4.1/?State=F
```

## Управление жестами

Кадр **зеркалится по горизонтали** (`cv2.flip(..., 1)`): «левая» / «правая» рука на экране совпадают с твоим ощущением «слева / справа».

### Левая рука — газ, стоп, назад

Учитываются **четыре пальца**: указательный, средний, безымянный, мизинец.  
Палец считается **поднятым**, если кончик **выше** среднего сустава (в координатах MediaPipe меньше `y`).

| Жест | Указательный | Средний | Безымянный | Мизинец | Результат |
|------|:---:|:---:|:---:|:---:|-----------|
| **Кулак** | вниз | вниз | вниз | вниз | **Стоп** |
| **Открытая ладонь** | вверх | вверх | вверх | вверх | **Вперёд** |
| **«Рога»** | вверх | вниз | вниз | вверх | **Назад** |
| Любая другая комбинация | — | — | — | — | **Стоп** |

«Рога» — жест как «рок»: только указательный и мизинец торчат, средний и безымянный согнуты.

### Правая рука — поворот при движении вперёд

Поворот работает, когда левая рука в режиме **«вперёд»** (открытая ладонь).  
Наклон ладони задаётся углом вектора от запястья (точка 0) к основанию среднего пальца (точка 9).

| Наклон | Результат |
|--------|-----------|
| Почти прямо (в пределах мёртвой зоны ±`STEER_DEAD_ZONE_DEG` градусов) | Просто вперёд |
| В одну сторону | Вперёд + влево (`FWD_L` → `G`) |
| В другую сторону | Вперёд + вправо (`FWD_R` → `I`) |

Если **правая рука не в кадре**, при открытой левой ладони машина едет **только вперёд** (`FWD`).

**Режим «назад»** (`REV`) с жеста «рога» **не комбинируется** с поворотом в текущей логике: назад — только прямо.

## Управление с клавиатуры

Используется **`cv2.waitKeyEx`**, чтобы на **Windows** корректно обрабатывались **стрелки** (у обычного `waitKey` они часто не приходят).

| Клавиша | Режим | `State` |
|---------|-------|---------|
| **W** или **стрелка вверх** | Вперёд | `F` |
| **S** или **стрелка вниз** | Назад | `B` |
| **A** или **стрелка влево** | Вперёд + влево | `G` |
| **D** или **стрелка вправо** | Вперёд + вправо | `I` |
| **Пробел** | Стоп | `S` |
| **Q** | Выход из программы | — |

**Приоритет:** пока активен режим клавиатуры, он **перекрывает** жесты. Активность сбрасывается через **`_KEY_TIMEOUT_SEC`** секунд после **последнего** распознанного нажатия (если не нажимать клавиши, снова работают только руки).

На HUD справа внизу: подсказка или строка вида **`KB: FWD`**, когда клавиатура перехватила управление.

## Интерфейс на экране (HUD)

Верхняя тёмная полоска поверх видео:

- **Левая колонка** — распознанный жест левой руки (вперёд / назад / стоп) и краткая подсказка по жестам.
- **Правая колонка** — поворот правой руки (влево / прямо / вправо) и угол наклона `tilt: ... deg`.
- **CMD:** — итоговый режим и **`State=`** буква, уходящая на ESP.
- **WiFi:** — краткий результат последнего HTTP-запроса (`ok`, `timeout`, `net err` и т.д.).
- Внизу справа — режим клавиатуры или подсказка по клавишам.

Центральная мини-графика: стрелка «руля» при повороте или индикатор **BACK** при движении назад.

## Протокол связи с машинкой

- Метод: **GET**
- URL: `http://<IP_ESP>/?State=<одна_буква>`
- Пример: `http://192.168.4.1/?State=S`

ESP в `HTTP_handleRoot` сохраняет значение `State` в переменную `command` и в цикле вызывает соответствующую функцию моторов. Запросы повторяются с интервалом **`CAR_WIFI_HEARTBEAT_SEC`**, чтобы при обрыве связи последняя команда не «зависала» навсегда без обновления (логика на стороне Python).

## Опционально: Serial

В `main.py` есть заготовка под COM-порт (`CAR_SERIAL_ENABLED`, `CAR_SERIAL_PORT`, `CAR_SERIAL_BAUD`). Сейчас отправка по Serial **отключена** (`False`). Если включишь и допишешь отправку байтов в порт, формат должен совпадать с тем, что ожидает твоя схема на Arduino (в текущей прошивке из репозитория управление идёт по **Wi‑Fi**, не по Serial из `main.py`).

## Настройка параметров в `main.py`

| Переменная | Назначение |
|------------|------------|
| `CAR_WIFI_ENABLED` | `True` / `False` — слать ли команды по Wi‑Fi |
| `CAR_WIFI_BASE_URL` | Базовый URL ESP, например `http://192.168.4.1` |
| `CAR_WIFI_HEARTBEAT_SEC` | Как часто повторять запрос при том же режиме |
| `CAR_WIFI_TIMEOUT_SEC` | Таймаут одного HTTP-запроса |
| `STEER_DEAD_ZONE_DEG` | Мёртвая зона руля (градусы), правая рука |
| `_KEY_TIMEOUT_SEC` | Сколько секунд держать «режим клавиатуры» после последнего нажатия |
| `VideoCapture(0)` | Номер камеры; при двух камерах попробуй `1`, `2` |

## Решение проблем

| Симптом | Что проверить |
|---------|----------------|
| Окно не открывается / нет картинки | Камера не занята другой программой; смени индекс `VideoCapture` |
| `hand_landmarker.task` not found | Файл в `models/hand_landmarker.task`, путь как в `MODEL_PATH` |
| WiFi: `timeout` / `net err` | IP в `CAR_WIFI_BASE_URL`, ноутбук в нужной сети, ESP включён, порт 80 |
| Машина не реагирует на жесты | Освещение, руки в кадре, чёткие жесты «кулак / ладонь / рога» |
| Клавиши не работают | Фокус на окне **Image**; на Windows нужен `waitKeyEx` (уже в коде) |
| Лево/право перепутаны | Кадр зеркалится специально; если неудобно — убери `cv2.flip` в коде и проверь снова |
| Жест «назад» срабатывает плохо | Выполни «рога» чётче: только указательный и мизинец вверх |

## Краткий чеклист перед первым запуском

1. Установлены зависимости: `pip install -r requirements.txt`
2. Есть файл `models/hand_landmarker.task`
3. ESP8266 прошит, Serial Monitor показывает IP точки
4. Ноутбук подключён к Wi‑Fi робота (или к той же сети, что ESP)
5. В `main.py` указан верный `CAR_WIFI_BASE_URL`
6. Запуск `python main.py`, фокус на окне с видео для клавиш
